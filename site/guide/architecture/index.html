    Architecture¶

    LEAF DSS is a monolithic Flask application serving both the web UI and REST API from a single process. This design simplifies deployment while keeping the codebase maintainable through clear module separation.

    Key Modules¶

    Module File Responsibility
    App app.py Flask routes (31+ endpoints), request handling, Flasgger Swagger config, MkDocs serving
    Config config.py Color palettes, variable group definitions, map center/zoom settings
    Data Utils data_utils.py Shapefile loading (block, GP, district, protected areas), GeoJSON conversion, variable metadata extraction, GP-block mapping
    Feasibility feasibility.py Weighted score calculation, classification into feasibility classes, distribution statistics
    RAG Utils rag_utils.py PDF processing, ChromaDB vector store management, OpenAI recommendation generation

    Data Flow¶

    Initial Page Load¶

    Browser → GET /
  └→ render_template(index.html) with interventions, config
Browser → GET /api/blocks (or /api/gp for GP level)
  └→ load_shapefile() → GeoDataFrame → GeoJSON
Browser → GET /api/districts/geojson
  └→ load_district_boundaries() → GeoJSON
Browser → GET /api/protected-areas/geojson (if toggled on)
  └→ load_protected_areas() → GeoJSON
Browser → Leaflet.js renders all layers

    Feasibility Calculation¶

    Browser → POST /api/calculate-feasibility
  └→ Parse { intervention, filters, logic, district } from JSON body
  └→ load_shapefile() → GeoDataFrame (cached)
  └→ If intervention only: load filters from get_intervention_config()
  └→ add_feasibility_to_gdf(gdf, filters, logic)
      └→ For each block: score = Σ(in_range × weight) / Σ(weight)
      └→ Classify scores into feasibility classes
  └→ calculate_and_get_geojson() → scored GeoJSON + distribution stats
  └→ Return JSON response
Browser → Update map colors, chart, and filter indicators

    AI Recommendation¶

    Browser → POST /api/ai-recommendation
  └→ Lazy-import rag_utils (first call only)
  └→ Build context string from block data + metrics
  └→ Query ChromaDB for relevant policy document chunks
  └→ Construct prompt: context + retrieved chunks + instruction
  └→ Send to OpenAI GPT
  └→ Return { recommendation, sources }

    URL-Based Navigation¶

    Browser → GET /Tinsukia/Digboi
  └→ render_app(level='block', district='Tinsukia', block='Digboi')
  └→ Frontend reads window.INITIAL_STATE
  └→ Applies district filter → zooms map → opens block detail

    Caching Strategy¶

    Data loading functions use @lru_cache(maxsize=1) to cache results in memory after first access:

    Function What's Cached Approx. Size
    load_shapefile() Block GeoDataFrame (152 rows) ~2 MB
    load_gp_shapefile() GP GeoDataFrame (95 rows) ~1 MB
    load_district_boundaries() District GeoJSON dict ~500 KB
    load_protected_areas() Protected areas GeoJSON dict ~5 MB
    load_metadata() Variable CSV DataFrame ~100 KB
    load_district_mapping() District ID→name dict ~1 KB
    load_gp_data() GP indicator DataFrame ~50 KB
    load_gp_block_mapping() GP name→block dict ~5 KB

    Cache is populated on first request and persists for the lifetime of the process. On Render, this means until the next deployment or dyno restart.

    Cache Invalidation

    There is no cache invalidation mechanism. If data files change, the server must be restarted. In development, the Flask debug reloader handles this automatically.

    Frontend Architecture¶

    The frontend is a single-page application (SPA) built with vanilla JavaScript:

    Component Technology File
    Map Leaflet.js 1.9.4 static/js/app.js
    Charts Chart.js 4.4.1 static/js/app.js
    Icons Bootstrap Icons 1.11.3 CDN
    Fonts Inter (Google Fonts) CDN
    Layout CSS Grid + Flexbox static/css/style.css

    Global State Object¶

    All application state is stored in a single state object (app.js:10):

    • state.map — Leaflet map instance
    • state.geojsonLayer — Current data layer (blocks or GPs)
    • state.districtLayer — District boundary overlay
    • state.protectedAreasLayer — Protected areas overlay
    • state.currentIntervention — Selected intervention name
    • state.currentFilters — Active filter array
    • state.currentLevel — 'block' or 'gp'
    • state.activeVariable — Variable for single-variable choropleth mode

    Map Layers (z-order, bottom to top)¶

    1. CartoDB Light tile layer (base map)
    2. state.geojsonLayer — Feasibility-colored blocks/GPs
    3. state.protectedAreasLayer — Green protected area polygons (optional)
    4. state.districtLayer — Dashed district boundaries

    API Route Organization¶

    Routes in app.py are organized by section (in order):

    1. Page Routes — HTML template rendering (/, /<district>, /<district>/<block>, /<district>/<block>/<gp>)
    2. Block APIs — GET /api/blocks/*
    3. Location APIs — GET /api/locations, /api/districts/*
    4. Protected Areas — GET /api/protected-areas/geojson
    5. Intervention APIs — GET /api/interventions, /api/intervention/<name>/config
    6. Variable APIs — GET /api/variables, /api/variable-groups, /api/variable-stats/*
    7. Feasibility APIs — POST /api/calculate-feasibility, POST /api/gp/calculate-feasibility
    8. AI APIs — POST /api/ai-recommendation, POST /api/ai-recommendation/init
    9. Export API — POST /api/export/csv
    10. Config APIs — GET /api/config, /api/levels
    11. GP APIs — GET /api/gp/*, POST /api/gp/calculate-feasibility
    12. Health & Meta — GET /health, GET /api
    13. Static Serving — /ai-docs/*, /documentation/*
    14. Error Handlers — 404 (returns main app for non-API routes), 500
