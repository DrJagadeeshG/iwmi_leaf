    Production Tool API¶

    Two-way integration with the external production tool that downstream partners are building. The contract per the IWMI requirements call (2026-04-23):

    • Outbound (we publish): finalised clusters with their villages, members, and assigned coordinators.
    • Inbound (they push): aggregated dashboard data per cluster (e.g. for a duckery cluster: eggs produced, meat output). Per-user filtering and authentication live in the production tool - only aggregates flow back to LEAF DSS.

    A cluster only appears in the outbound feed once it has been finalised - typically after the CSV edit cycle has stabilised the village list.

    Finalise a Cluster¶

    POST /api/clusters/<cluster_id>/finalize

    Sets the cluster's finalized flag. Pass {"finalized": false} to revert (e.g. if further edits are needed).

    Body¶

    { "finalized": true }

    finalized defaults to true if the body is omitted.

    Response¶

    {
  "cluster_id": "KHOWANG-Goatery-39e1bdee",
  "finalized": true,
  "...": "rest of cluster fields"
}
    Code Description
    200 Updated cluster record.
    404 Cluster ID not found.

    Outbound Clusters Feed¶

    GET /api/production-tool/clusters

    Read-only list of finalised clusters. The production tool calls this whenever it needs to refresh its cluster catalogue. Same scope filters as /api/clusters.

    Query Parameters¶

    Param Type Required Description
    block string No
    district string No
    commodity string No One of the six commodities.

    Response¶

    [
  {
    "cluster_id": "KHOWANG-Goatery-39e1bdee",
    "commodity": "Goatery",
    "block_name": "KHOWANG",
    "district_name": "DIBRUGARH",
    "total_members": 49,
    "max_span_km": 1.193,
    "centroid_lat": 27.204,
    "centroid_lon": 94.808,
    "villages": [ /* ... */ ],
    "pashu_sakhi": "Smt. R. Devi",
    "block_coordinator": "Sri B. Sarma",
    "finalized": true
  }
]

    Cluster Production Dashboard¶

    GET  /api/production-tool/dashboard/<cluster_id>
POST /api/production-tool/dashboard/<cluster_id>

    POST accepts an arbitrary JSON payload and stores it on the cluster under dashboard. GET returns the last stored payload (or {} if none). Schema is intentionally open - the production tool sends whatever its commodity dashboard produces; LEAF DSS surfaces it on the cluster report card without inspection.

    Example POST Body¶

    {
  "period": "2026-Q1",
  "eggs_produced": 12450,
  "meat_kg": 320.5,
  "active_members": 42
}

    Response¶

    For both GET and POST, returns the payload object directly.

    Code Description
    200 Payload returned (empty object if not yet posted).
    400 POST without a JSON body.
    404 Cluster not found.

    Example¶

    # Push aggregates
curl -X POST https://iwmi-leaf.onrender.com/api/production-tool/dashboard/KHOWANG-Duckery-abc12345 \
     -H "Content-Type: application/json" \
     -d '{"period":"2026-Q1","eggs_produced":12450,"meat_kg":320.5}'

# Read latest
curl https://iwmi-leaf.onrender.com/api/production-tool/dashboard/KHOWANG-Duckery-abc12345
